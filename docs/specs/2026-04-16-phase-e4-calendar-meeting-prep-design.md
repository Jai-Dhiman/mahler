# Phase E4 — Google Calendar Skill + Intelligent Meeting Prep System

**Goal:** Mahler can list and create Google Calendar events via Discord, proactively delivers an intelligent prep brief roughly one hour before meetings, and stays passively aware of upcoming meetings during all chat turns.

**Not in scope:**
- Event update and delete operations
- Post-meeting extraction (action item capture after the meeting ends)
- Multi-calendar support (primary calendar only)
- Outlook or any non-Google calendar integration
- Meeting notes or agenda creation

---

## Problem

Without this, Mahler has no knowledge of the user's schedule. It cannot answer "what do I have today?", cannot schedule events via Discord, and has no way to surface relevant context before an important call. The user must context-switch to Google Calendar manually and prepare for meetings without the email/task/wiki context Mahler already holds.

---

## Solution (from the user's perspective)

- Ask Mahler "what do I have tomorrow?" → formatted list of calendar events.
- Say "schedule a call with Alice at 3pm Friday" → Mahler creates the event and confirms.
- One hour before any meeting, a Discord brief appears with: meeting metadata (title, attendees, time), recent flagged emails from the attendees (last 7 days, URGENT/NEEDS_ACTION from `email_triage_log`), open Notion tasks, and an optional wiki lookup for non-trivial meeting topics. The brief is synthesized by the LLM — not a raw data dump.
- During normal Discord chat within two hours of a meeting, Mahler passively notes the upcoming meeting in its context so it naturally factors it into responses.

---

## Design

### Calendar skill: script-first

`google-calendar` follows the established pattern (`gmail_client.py` → `gcal_client.py`, `tasks.py` → `gcal.py`, `SKILL.md`). The CLI exposes `list` and `create`. All Calendar API complexity — token refresh, pagination, datetime normalization, attendee extraction — is hidden in `gcal_client.py`. `SKILL.md` tells Hermes when and how to invoke the CLI and how to convert relative dates to ISO 8601.

### Meeting prep: Hermes-native cron

Rather than a monolithic Python pipeline (like `email-triage`'s `triage.py`), the meeting prep brief is generated by a Hermes cron job that uses existing skills organically. A Hermes cron fires every 15 minutes. Its SKILL.md instructs Hermes to: fetch calendar events, check deduplication, gather context from email/tasks/wiki, synthesize the brief, post to Discord, and log completion.

This approach was chosen over a script-first pipeline because the intelligence step — deciding which wiki concepts are relevant, which emails are meaningful, how to synthesize — is naturally LLM work. The deterministic parts (D1 check/log, email context query) are scripted; the synthesis is Hermes's.

### Deduplication: window trick + D1 log

The cron checks for events starting 45–75 minutes from now. This 30-minute window means any 15-minute poll fires inside the window exactly once per meeting. Before generating a brief, `dedup.py check` queries `meeting_prep_log` — if the event_id is already there, it exits 1 and Hermes stops. After posting, `dedup.py log` writes the row. The check command raises `RuntimeError` on D1 errors rather than silently returning "not seen", which would cause duplicate briefs.

### Ambient awareness: pre_llm_call plugin

A Hermes plugin fires before every LLM turn. It queries `meeting_prep_log` for any logged meeting starting within 2 hours and injects a one-line context string ("Upcoming meeting in 45min: Budget review"). The plugin returns `None` — silently — on any exception. This is the only intentional silent-failure in the system; a D1 hiccup must never break a normal chat turn.

## Modules

### `gcal_client.py`

- **Interface:** `refresh_access_token(client_id, client_secret, refresh_token) -> str`, `list_events(access_token, time_min, time_max, max_results=50) -> list[dict]`, `create_event(access_token, summary, start, end, attendees=None, description=None) -> dict`
- **Hides:** OAuth2 token refresh via `https://oauth2.googleapis.com/token`, Google Calendar Events API pagination, datetime normalization (both `dateTime` and `date` fields to a single ISO string), attendee list extraction from nested `attendees[].email` structure, HTTPS opener construction with cert verification.
- **Depth:** DEEP — simple 3-function interface conceals all Calendar API protocol.

### `gcal.py`

- **Interface:** CLI subcommands `list [--days N | --hours-ahead N]` and `create --title TITLE --start ISO --end ISO [--attendees emails] [--description TEXT]`
- **Hides:** env loading (hermes `.env` supplement), argument parsing, client construction, output formatting, relative-date conversion instructions delegated to SKILL.md.
- **Depth:** DEEP — 2-command CLI interface hides all plumbing; mirrors `tasks.py` exactly.

### `meeting-prep/d1_client.py`

- **Interface:** `D1Client(account_id, database_id, api_token)` with `ensure_meeting_prep_table() -> None`, `is_already_notified(event_id: str) -> bool`, `insert_meeting_prep(event_id, summary, start_time) -> None`
- **Hides:** Cloudflare D1 REST API, HTTP request construction, JSON encoding, error parsing, table creation SQL.
- **Depth:** DEEP — same depth as `email-triage/d1_client.py`; meeting-prep-specific methods only.

**`meeting_prep_log` D1 schema:**
```sql
CREATE TABLE IF NOT EXISTS meeting_prep_log (
    event_id    TEXT PRIMARY KEY,
    summary     TEXT NOT NULL,
    start_time  TEXT NOT NULL,
    notified_at TEXT NOT NULL
)
```
`notified_at` is set to `datetime('now')` on insert. `start_time` is an ISO 8601 string (`2026-04-16T15:00:00Z`). The plugin queries this table with `WHERE start_time > datetime('now') AND start_time < datetime('now', '+2 hours')` ordered by `start_time ASC LIMIT 1`.

### `dedup.py`

- **Interface:** CLI `check --event-id ID` (exit 0 = proceed, exit 1 = already notified, non-zero + RuntimeError = D1 failure) and `log --event-id ID --summary TEXT --start-time ISO8601`
- **Hides:** D1 client construction, env loading, `ensure_meeting_prep_table` on every run, exit code semantics.
- **Depth:** DEEP — two-command CLI with meaningful exit codes hides all D1 interaction.

### `email_context.py`

- **Interface:** CLI `email-context --attendees "email1@x.com,email2@x.com"`
- **Hides:** D1 query against `email_triage_log`, domain extraction from attendee emails, formatting.
- **Depth:** DEEP — one command hides cross-table D1 query logic.

### `calendar-aware/plugin.py`

- **Interface:** `register(ctx)` (Hermes plugin entrypoint); `upcoming_meeting_context(session_id, user_message, is_first_turn, _now=None, **kwargs) -> dict | None`
- **Hides:** D1 query for upcoming meetings, time-until calculation, `~/.hermes/.env` loading, all exception handling (always returns None on failure).
- **Depth:** DEEP — one callable hides all state lookup and always produces safe output.

## Open Questions

- Q: Should all-day events (birthdays, holidays) be filtered out of the meeting prep cron?
  Default: Yes — filter out events where `start` is a date-only string (no `T` separator). These are never meetings.
- Q: Should the wiki lookup be attempted for every meeting or only when the meeting title contains non-trivial keywords?
  Default: Skip wiki lookup when meeting title is a person's name only (e.g. "1:1 with Alice"), a catch-all ("sync", "standup", "catch up"), or has no description. Otherwise attempt one search query using the meeting title + first 50 chars of description.
